Audit Document

Canopy reorder math, audited side-by-side with the WDS Purchasing Reorder Report

A field-by-field comparison of how Canopy generates its reorder recommendations versus the legacy WDS process — same SKUs, same data, two very different outputs. Built for Golf to audit the calculation logic; designed so leadership can see why the upgrade matters.

Prepared for: Winsome purchasing & finance leadership Worked example: Item #94918 (CLAIRE Accent Table) As of: May 2026
01 · At a glance

One report vs one dashboard

Today, a buyer reviewing 600 SKUs cycles through three WDS screens per item — Reorder Report, F5 sales history, Shift F9 projected inventory — and applies judgment to merge the three views. Canopy collapses that loop into one calculation that ingests the same WDS data plus Amazon's consumer-side signal.

📋 Current process — WDS
3 screens, manual
1
Run Purchasing Reorder Report
Per-Agent run with selection inputs: Backup Inventory months, Transit Time, Reorder Window. Output is one row per SKU with Reorder ETD.
2
F5 — 2nd & 3rd Inventory Inquiry screens
Per-item: scroll through 12+ months of unit sales, returns, invoice $; then channel split (#9944 1P, #1750 DF, OTHRS, #2200 DI). Eyeball trends, year-over-year swings, Amazon spikes. Talk to manager when fluctuations aren't explainable from the numbers alone.
3
Shift F9 — Projected Inventory
Per-item: ledger of customer/transfer orders + scheduled receipts (open POs with ETA), month by month. Used to push the suggested ETD later if open POs would arrive too close together.
4
Buyer adjusts the Reorder ETD
Final answer is the Reorder Report's ETD ± buyer's manual judgment from steps 2 and 3. Buyer must also mentally subtract production lead time to figure out the actual PO-placement date.
⚡ Canopy process
1 dashboard, automated
1
Velocity (auto)
Weighted blend of 3-mo / 6-mo / 12-mo windows (50% / 30% / 20%) with kit-implied units rolled in and a per-month seasonality factor applied.
2
Inventory at arrival (auto)
Woodinville on-hand + Amazon FC on-hand + open POs arriving before lead-time cutoff − projected demand during lead time. F9-style logic baked into the math.
3
Tier-based targets (auto)
Each SKU has a tier (A / B / C / Low Priority) with its own target days of supply, safety days, and Amazon DOI target. No single global "backup months" — high-revenue SKUs get more buffer than tail items.
4
Reorder qty + decision + order-by date (auto)
Quantity respecting MOQ. Decision: order / watch / do_not_order. Date = projected stockout − full lead time (production + transit + port). Container-fit hint included.
5
Amazon DOI + forecast variance + pull-back alert (auto)
Combined Amazon FC + Woodinville pool sized against Amazon's own forward forecast. Surfaces the SKUs where Amazon's algorithm is signaling a pull-back before it shows up in WDS shipment history.
The three screens are still useful — Canopy isn't replacing buyer judgment, it's eliminating the busywork that wraps around it. Buyers still review surfaced edge cases; the Q4 spikes that don't repeat are exactly what Canopy promotes to the top of the queue.
02 · The math

Same inputs, different outputs

The WDS Reorder Report uses an arithmetic mean of three velocity windows. Canopy uses a weighted blend, applies seasonality, separates target inventory from safety stock, and back-calculates the order-by date using the full lead time.

WDS formula
audit reference 
VelocityMONTHLY_SALES = (v_3mo + v_6mo + v_12mo) / 3
   // arithmetic mean of three windows

Backup bufferBACKUP_QTY = MONTHLY_SALES × backupMonths
   // backupMonths set per report run (e.g. 1.50)

CoverageGOOD_FOR_MO = (ONHAND + IMPORT_ON_ORDER − BACKUP_QTY) / MONTHLY_SALES
   // months of coverage above the backup floor

Factory ready dateREORDER_ETD = today + GOOD_FOR_MO × 30.42 − TRANSIT_DAYS
   // only subtracts transit; buyer mentally
   // subtracts production lead time

Filter flag* if REORDER_ETD ≤ today + REORDER_WINDOW
   // asterisk on rows that need attention soon
Canopy formula
implemented 
Step 1 — velocityv_3mo  = sum(units in 13 weeks)  / 13
v_6mo  = sum(units in 26 weeks)  / 26
v_12mo = sum(units in 52 weeks)  / 52
blendedWeekly = (v_3mo×0.5 + v_6mo×0.3 + v_12mo×0.2)
   // + kit-implied units (parent×qtyPerKit)
forecastVelocity = blendedWeekly × seasonality[month]

Step 2 — inventoryonHand     = Woodinville + Amazon FC
inbound    = Σ(qtyOrdered − qtyReceived) for open POs
inventoryAtArrival = onHand + arrivingByCutoff −
                     round(forecastVelocity × leadDays/7)

Step 3 — targets per tierrequired = round(forecastVelocity × (tier.targetDays+tier.safetyDays)/7)

Step 4 — qty & decisionrawReorderQty = required − inventoryAtArrival
adjustedQty   = max(rawReorderQty, MOQ)
decision = order | watch | do_not_order

Step 5 — timing (full lead time)recommendedOrderByDate = projectedStockoutDate
                       − totalLeadTimeDays
   // includes production + transit + port
Subtle but important difference. WDS's Reorder ETD is the date goods need to leave the factory (factory-ready). The buyer must mentally subtract another ~60 production days to figure out the actual PO-placement date. Canopy's recommendedOrderByDate already subtracts the full lead time and gives the buyer the date they should be sending the PO.

04 · What Canopy adds

Capabilities not present in the WDS Reorder Report

The base WDS report does one thing well — it tells you when stock will run out at current blended velocity. Canopy keeps that calculation and layers on the signals buyers currently piece together by hand or do without entirely.

Inventory & demand
  • Amazon FC + Woodinville pool. WDS reports Woodinville only. Canopy adds Amazon's FC inventory to the available pool — material when Amazon holds significant stock for 1P / DI.
  • Weighted velocity blend. 50% / 30% / 20% across 3-mo / 6-mo / 12-mo, vs WDS's flat arithmetic mean. Recent shifts get more weight.
  • Per-month seasonality factor. Buyer-set; default 1.0. WDS has no seasonality.
  • Kit-implied demand. Child SKUs automatically inherit demand from any Parent kit they're part of (parent units × qty per kit). WDS treats kit Parents as standalone SKUs.
  • Open POs auto-rolled in. F9-style projected inventory baked into the math, filtered to POs arriving within the lead-time window.
Decision quality
  • Tier-aware targets. A / B / C / Low Priority each get distinct target & safety days. WDS uses one global "backup months" per report run.
  • Order-by date (full lead time). Subtracts production + transit + port processing. WDS only subtracts transit; buyer mentally adds production.
  • Three-state decision. order / watch / do_not_order. WDS has a single asterisk if within reorder window.
  • FCL container hint. Per-SKU 40HQ fraction with lcl / round_up_to_fcl / fcl / multi_fcl guidance. WDS has none.
  • Amazon DOI + forecast variance + pull-back alerts. Read Amazon's forward signal alongside Winsome's history. WDS has no Amazon-side data.

05 · Time impact

Buyer hours per review cycle

The biggest leadership win isn't a sharper formula — it's the hours back. WDS's three-screen review takes 2–5 minutes per SKU; Canopy bulk-processes the catalog and surfaces only the SKUs needing attention.

WDS
~30 hours per review · 600 SKUs × ~3 min each
Canopy
~3 hours · review surfaced edge cases only

Canopy's 3 hours = buyer time spent on SKUs Canopy promotes to the queue (decision = order or watch with anomalies). The 90% of SKUs in steady state are reviewed automatically in the bulk run. Both estimates assume one buyer doing the full catalog.

06 · Trade-offs

Honest comparison

Both systems have legitimate strengths. WDS is the proven status quo; Canopy is the upgrade with a real (small) implementation cost. Naming both keeps this audit credible.

📋 WDS Reorder Report

Strengths

  • Proven, deeply familiar to buyers and Kasey
  • No external integration risk — runs on existing WDS data
  • F5 + F9 give per-item judgment loop where buyer expertise lives
  • Per-Agent grouping matches purchasing org structure

Limitations

  • Manual three-screen scan: ~3 min × 600 SKUs = 30 buyer hours per review
  • Single arithmetic-mean blend is bursty when 1P POs spike (94918 example)
  • No seasonality, no kit-implied demand, no Amazon-side signal
  • Reorder ETD is factory-ready, not order-by — buyer mentally subtracts production lead
  • One global backup-months setting per run — no tier differentiation
  • Amazon FC inventory invisible — pool size understated for 1P SKUs
⚡ Canopy

Strengths

  • Single dashboard collapses Reorder Report + F5 + F9 review
  • Weighted velocity blend + per-month seasonality
  • Tier-aware targets (A / B / C / Low Priority)
  • Order-by date subtracts the full lead time
  • Amazon DOI + forecast variance + pull-back alerts (V2)
  • Kit-implied demand auto-computed
  • FCL container hint per SKU
  • ~90% time reduction on routine review

Limitations

  • New tool — buyer training curve, change management
  • Depends on Golf's nightly data feeds being reliable
  • Amazon side currently manual until Golf's Amazon API is live
  • parthist.json doesn't yet expose customer account #, so 1P / DF / Domestic split inside LOC 1 is blended (Golf follow-up in flight)
